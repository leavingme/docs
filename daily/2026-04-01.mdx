### Claude Code「烧Token狂魔」元凶曝光：autoCompact失败后无限重试，有会话连续失败3272次

**CatFly (@imyouhu)**

**[泄露源码挖出的最实用成果]**

有人在拿泄露源码丢给 Codex 分析后，定位到了 Claude Code 疯狂消耗 token 的罪魁祸首——autoCompact（自动上下文压缩）机制在失败后会无限重试，完全没有上限。源码注释记录曾有会话连续失败高达 3272 次。

**影响：** "用两下就触发限速"的抱怨，很大一部分源于此。这暴露了 Claude Code 作为产品而非纯研究项目，其工程债务的冰山一角。

### Anthropic埋的三层反蒸馏护城河：泄露源码完整曝光了这项「防竞争对手训练」的隐秘工程

**宝玉 (@dotey)**

**[Claude Code泄露源码最有战略价值的分析]**

鸭哥分析指出 Anthropic 在 Claude Code 里埋了三层反蒸馏机制：

**第一层：输出掺假。** API返回结果时服务端混入虚假工具调用数据。正常用户不受影响（服务端过滤），但批量抓取API输出训练模型的竞争对手，会把这些假数据吃进训练集，污染模型质量。

**第二层：推理过程加密。** Claude工具调用间的中间推理文本（"我先读这个文件，再检查语法"）对蒸馏训练价值极高。这一层将其替换为加密签名，下一轮对话客户端拿签名换回原文。外部只能看到摘要，完整推理链不可得。

**第三层：协议隔离。** Claude Code用独立JSON协议格式和版本标记与API通信，与每周920万次普通API请求在统计上隔离开，服务端可对不同群体差异化处理。

**影响：** 这是理解大厂AI战略博弈的核心——模型厂商与竞争对手之间的「数据战争」已经工程化、体系化。

gakki锐评：蒸馏战争从「道德争议」升级为「工程军备竞赛」。护城河不是想法，是代码。

### 字节已死，token当立：Karpathy「LLM = CPU，Agent = OS内核」的范式级隐喻

**Orange AI (@oran_ge)**

**【从牛顿力学到量子力学：计算的底层正在易主】**

Karpathy 给出了一个被低估的类比：LLM = CPU，Agent = 操作系统内核。但他补了一个关键细节——LLM 处理的数据单位不是字节，是 token；运行方式不是确定性，而是统计性。

这意味着什么：整个数字世界过去60年建立在「字节 + 确定性」之上，精确、不会出错。现在 LLM 引入了 token + 概率性，同一输入跑两遍可能不同结果。

这不是性能提升，是底层计算范式的转移——从牛顿力学进入量子力学。

**gakki锐评：** 大多数人还在讨论「AI 能不能替代 X」，但真正重要的问题是：当基本单位从字节变成 token，现有的软件架构、数据库、安全模型全都要重建。这个切换的窗口期，比想象中近。

### 宝玉解码Claude Code闭源逻辑：隐藏功能、迭代自由与「愚人节彩蛋」的工程理性

**宝玉 (@dotey)**

**[一句话核心判断]**

**信息增量：**
• 闭源可以暗搓搓加料：防蒸馏、用户标识追踪、甚至「不小心」导致第三方 prompt caching 失效的 bug

• 闭源迭代无需走复杂代码审查流程，迭代速度是开源的数倍

**gakki 锐评：**
宝玉的分析戳破了一个幻觉：开源 vs 闭源的之争本质是「透明度」vs「灵活性」的权衡，而非「正义」vs「邪恶」。对于商业公司而言，闭源的保护的是迭代自由和合规弹性，而非代码本身的价值。这个视角对评估 Agent 工具的商业前景有重要参考意义。
